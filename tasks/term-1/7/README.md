# Параллельное умножение матриц на Java

## Сравнение производительности

Результаты бенчмаркинга на процессоре Intel Core i5-12500H (Windows 11), 5 прогонов для каждого размера:

| Размер матриц | Однопоточно, мс | Многопоточно (24 потока), мс | Ускорение |
|---------------|-----------------|------------------------------|-----------|
| 1000×1000     | 590–640         | 125–140                      | 4.6–4.8×  |
| 1500×1500     | 2800–3000       | 520–580                      | 5.0–5.4×  |
| 2000×2000     | 6000–6400       | 1350–1450                    | 4.4–4.7×  |

Для матриц размером менее 400×400 автоматически применяется однопоточный режим, так как накладные расходы на создание и синхронизацию потоков превышают выигрыш от параллелизации.

## Выбор оптимального количества потоков

Процессор Intel Core i5-12500H имеет гибридную архитектуру:
- 12 физических ядер (4 производительных P-cores + 8 энергоэффективных E-cores)
- 16 логических процессоров (гипертрединг только на P-cores)

Для выбора оптимального количества потоков проведён эмпирический бенчмаркинг на матрицах 1500×1500 с 10 прогонами для каждого варианта:

| Количество потоков | Среднее время, мс | Ускорение относительно однопоточного режима |
|--------------------|-------------------|---------------------------------------------|
| 12                 | 585               | 4.9×                                        |
| 16                 | 550               | 5.2×                                        |
| 24                 | 535               | 5.4×                                        |
| 32                 | 560               | 5.1×                                        |

Оптимальное значение 24 потока объясняется следующими факторами:

1. Более равномерное распределение строк матрицы между потоками (62–63 строки на поток для матрицы 1500×1500), что снижает время ожидания завершения самого медленного потока.

2. Балансировка нагрузки между производительными (P-cores) и энергоэффективными (E-cores) ядрами гибридной архитектуры.

3. Сокрытие задержек кэш-памяти за счёт дополнительных потоков, выполняемых при кэш-промахах.

Значение 24 выбрано эмпирически как точка максимума производительности. При увеличении количества потоков свыше 24 наблюдается деградация из-за избыточных переключений контекста операционной системой.
